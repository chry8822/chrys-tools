# AI 수정 행동 규칙

## RULE 5: 단계적으로 진행하라

**사용자가 단계별 확인을 원할 때:**
- 1단계씩 진행, 각 단계 완료 후 사용자 확인 필수
- 확인 없이 다음 단계 선진행 금지
- 형식: `[1/N 단계] 내용 → 완료. 다음 단계로 진행할까요?`

**사용자가 전체를 한 번에 요청한 경우:**
- 진행 전 영향받는 파일 목록과 작업 순서를 먼저 보고
- 보고 후 전체를 순서대로 진행

## RULE 6: 변경 전 반드시 사전 고지하라

파일 수정 전 아래를 먼저 보고한다:

```
[변경 대상 파일] 목록
[변경 이유] 분석 근거
[작업 순서] 1 → 2 → 3
[예상 영향] 변경으로 인해 바뀌는 동작
```

파일명 rename 또는 import 경로 변경 시 추가 절차:
1. 해당 파일을 import하는 모든 파일을 프로젝트 전체에서 검색
2. 참조 파일 목록 출력
3. 파일 rename + 모든 참조 import 경로 동시 수정
4. `npx tsc --noEmit` 로 잔여 에러 검증

참조 파일 검색 없이 파일명만 변경하는 것은 절대 금지.

## RULE 8: 기존 코드의 의도를 먼저 파악하라

수정 전 "왜 이렇게 작성되어 있는가"를 먼저 분석한다.
이상해 보이는 코드는 수정 전 의도를 질문하거나 아래 파일을 읽는다:
- eslint.config.js / .eslintrc
- tsconfig.json
- package.json

## RULE 9: 새 파일/함수 생성 전 기존 코드베이스를 검색하라

새 유틸/훅/상수/타입 생성 전 아래 경로에서 중복 존재 여부 검색:
- src/utils/ / src/hooks/ / src/constants/ / src/types/ / src/components/@shared/

검색 없이 새 파일 생성 금지. 중복 코드는 기술 부채다.

## RULE 12: 동일 파일 2회 수정 후 해결 안 되면 멈춰라

같은 파일에 같은 목적으로 2번 이상 수정했는데도 해결 안 되면:
1. 즉시 수정 중단
2. RULE 11 기준으로 데이터 흐름 전체 재추적
3. "다른 파일이 원인"이라는 가정으로 분석 재시작
4. 재분석 결과 보고 후 사용자 컨펌 받고 재개

## RULE 13: 작업 중 문제 발생 시 즉시 중단하고 컨펌 받아라

아래 상황 발생 시 즉시 멈추고 보고한다:
- 예상 외 타입/린트 오류 발생
- 수정 범위가 고지한 것보다 커질 것으로 판단
- 다른 파일에 사이드이펙트 가능성 발견
- 원인 분석이 불완전한 상태에서 계속이 위험하다고 판단

```
[중단 이유] 발생한 문제 또는 리스크
[현재 상태] 어디까지 진행했는가
[선택지] 방법 A / 방법 B
[질문] 어떻게 진행할까요?
```

컨펌 없이 임의로 계속 진행 금지.
