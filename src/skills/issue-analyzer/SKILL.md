# Issue Analyzer & Fixer Skill

Jira 티켓을 분석해 **빠르게 수정 가능한 이슈는 자동으로**, **복잡한 이슈는 사용자와 협업해서** 코드를 수정하고 테스트까지 실행합니다. 목표는 정확도를 유지하면서 수정 속도를 높이는 것입니다.

---

## 모델 라우팅 원칙

| 작업 | 모델 |
|------|------|
| Jira 조회, 코드 검색, Git 탐색, 테스트 실행 | `claude-haiku-4-5-20251001` |
| 복잡도 판단, 분석 보고서, 수정 계획, 코드 수정 | `claude-sonnet-4-6` (메인) |

서브에이전트는 **요약된 결과만 반환**합니다. raw JSON, 파일 전체 내용 전달 금지.

---

## 실행 순서

### 1단계: 티켓 조회 + 링크 탐색 + 복잡도 판단

티켓 번호를 파싱하고, 티켓 본문과 댓글의 **링크까지 전부 열어서** 내용을 파악한 뒤 복잡도를 판단합니다.

#### 1-A. 티켓 조회 (Haiku 서브에이전트)

- Atlassian MCP(`getJiraIssue`)로 조회. MCP 없으면 `~/.claude/config.json`의 baseUrl + apiToken으로 REST API 직접 호출
- 반환: 제목, 버그 설명, 재현 절차, 에러 메시지, 언급된 파일/API/함수명, 키워드 5개
- **링크 목록도 함께 추출**: 본문과 댓글에 포함된 모든 URL

#### 1-B. 링크 컨텍스트 수집 (Haiku 서브에이전트, 1-A와 병렬 실행 불가 — 1-A 완료 후 실행)

1-A에서 추출한 링크를 **전부 열어서** 내용을 확인합니다.

**Confluence 링크** (`*.atlassian.net/wiki/*`):
- `getConfluencePage` MCP로 페이지 내용 조회
- 핵심 내용 요약: 변경된 API 필드, deprecated 항목, 새 엔드포인트, 마이그레이션 가이드

**외부 링크** (GitHub, 사내 문서 등):
- `WebFetch`로 내용 조회
- 핵심 내용 요약

**반환 형식 (요약만):**
```
링크 컨텍스트:
1. [링크 제목] (Confluence/외부)
   - 핵심 내용: ...
   - deprecated 항목: ... (있을 경우)
   - 새 엔드포인트/필드: ... (있을 경우)
   - 수정에 필요한 정보: ...
```

> 링크가 없거나 열 수 없으면 이 단계는 건너뜁니다.

#### 1-C. 복잡도 판단 (메인 모델 — 티켓 요약 + 링크 컨텍스트 기반)

아래 신호가 **하나라도** 있으면 → **Fast Lane**
- 에러 메시지, 스택 트레이스가 포함됨
- 특정 파일명, 함수명, API 엔드포인트가 언급됨
- 댓글 또는 링크 문서에 수정 방향이 명시됨
- 링크 문서에서 deprecated API 또는 새 API 스펙이 확인됨

위 신호가 **전혀 없으면** → **Collaborative Lane**

---

### Fast Lane (빠른 자동 처리)

> 이슈 위치가 명확한 경우. 사용자 개입 없이 자동으로 진행합니다.

#### F-1단계: 정밀 탐색 (병렬, Haiku 서브에이전트 3개 동시 실행)

> 1단계에서 수집한 링크 컨텍스트(새 API 필드, deprecated 엔드포인트 등)를 탐색 키워드에 반드시 포함합니다.

**서브에이전트 1 — 에러/API 기반 탐색**
- 티켓의 에러 메시지, API명, 함수명 + **링크 문서에서 확인된 새 필드명/엔드포인트**로 Grep 검색
- deprecated API가 확인된 경우 → 해당 API를 현재 코드에서 사용하는 위치 탐색
- 반환: 일치하는 파일:라인 목록 (최대 5개), 각 1줄 요약

**서브에이전트 2 — Git 히스토리 탐색**
- `git log --oneline --all --grep="[티켓번호]"` 실행
- 관련 파일 최근 변경 이력 확인
- 반환: 관련 커밋 목록, 최근 변경 파일 (최대 3개)

**서브에이전트 3 — 연관 파일 추적**
- 서브에이전트 1이 찾은 파일의 import/dependency 1단계만 추적
- 반환: 직접 연관 파일 목록 (최대 3개), 각 1줄 요약

#### F-2단계: 수정 계획 제시 및 승인

탐색 결과를 바탕으로 수정 계획을 제시합니다:

```
## [티켓번호] 수정 계획

### 원인
[한 문장으로 요약]

### 수정할 파일
1. src/path/file.ts (line 42-58)
   - 변경 내용: [구체적으로]
   - 이유: [왜]

### 예상 사이드 이펙트
- [영향받을 수 있는 부분]

### 실행할 테스트
- [테스트 명령어]

위 계획대로 진행할까요?  [Y] 진행  [N] 취소  [E] 수정
```

- **Y / "응" / "ㅇㅇ"** → F-3단계
- **N** → 종료
- **E / 수정 요청** → 계획 수정 후 재제시

#### F-3단계: 코드 수정

- 수정 파일의 **관련 라인만** Read (파일 전체 읽기 금지)
- Edit으로 최소 범위만 수정
- 완료 후 1줄 요약 출력

#### F-4단계: 테스트 실행 (Haiku 서브에이전트)

프로젝트 루트에서 자동 감지:
1. `package.json` `scripts.test` → `npm test -- [수정파일 관련 패턴]`
2. `pytest.ini` / `pyproject.toml` → `pytest [관련 경로]`
3. `go.mod` → `go test ./[관련 패키지]/...`
4. 감지 실패 → 사용자에게 명령어 요청

가능하면 **전체 테스트가 아닌 관련 테스트만** 실행합니다.

결과:
- **PASS** → 완료 보고
- **FAIL** → 원인 분석 후 F-2단계부터 재시도 (최대 2회). 2회 초과 시 사용자에게 상황 보고 후 협업 경로로 전환

---

### Collaborative Lane (협업 처리)

> 이슈 위치가 불명확한 경우. 사용자의 힌트로 탐색 범위를 좁혀가며 진행합니다.

#### C-1단계: 1차 탐색 + 질문

키워드로 빠른 1차 탐색을 수행한 뒤, **구체적인 질문**으로 범위를 좁힙니다.

1차 탐색 (Haiku 서브에이전트):
- 티켓 키워드로 Grep + 최근 변경 파일 확인
- 반환: 유력 후보 파일 목록 (최대 3개) + 확신도(높음/중간/낮음)

탐색 결과를 보여주고 **예/아니오로 답할 수 있는 질문** 1~3개를 합니다:

```
## 1차 탐색 결과

유력 후보:
1. src/services/careService.ts — 간병 서비스 관련 로직 (확신도: 중간)
2. src/screens/TradeInfo.tsx — 거래정보 화면 (확신도: 낮음)

아직 확신이 없어서 몇 가지 여쭤볼게요:

Q1. 이 버그가 특정 사용자에게만 발생하나요, 모든 사용자에게 발생하나요?
Q2. 거래정보 화면에서 사용하는 API 이름이나 엔드포인트를 알고 있나요?
Q3. 최근 이 화면 관련 작업을 한 팀원이 있나요?
```

질문은 **범위를 좁히는 데 직접적으로 도움이 되는 것**만 선택합니다.
사용자가 답하면 → C-2단계로 이동합니다.

#### C-2단계: 힌트 기반 정밀 탐색

사용자의 답을 반영해 탐색 범위를 좁혀서 다시 검색합니다 (Haiku 서브에이전트).

- 힌트에서 새로운 키워드/파일명/API명 추출
- 정밀 Grep + 의존성 1단계 추적
- 반환: 재탐색 결과

결과에서 **위치 확신도가 높으면** → Fast Lane F-2단계로 합류해서 수정 진행
**여전히 불명확하면** → C-1단계 반복 (최대 3라운드)

3라운드 후에도 못 찾으면:
```
3번 탐색했지만 정확한 위치를 찾지 못했습니다.
지금까지 확인한 내용:
- [탐색한 파일 목록]
- [제외할 수 있는 영역]

직접 수상한 파일이나 함수를 알려주시면 바로 확인하겠습니다.
```
