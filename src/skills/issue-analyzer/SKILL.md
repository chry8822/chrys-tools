# Issue Analyzer & Fixer Skill

Jira 티켓을 분석해 **빠르게 수정 가능한 이슈는 자동으로**, **복잡한 이슈는 사용자와 협업해서** 코드를 수정하고 테스트까지 실행합니다. 목표는 정확도를 유지하면서 수정 속도를 높이는 것입니다.

---

## 모델 라우팅 원칙

| 작업 | 모델 |
|------|------|
| Jira 조회, 코드 검색, Git 탐색, 테스트 실행 | `claude-haiku-4-5-20251001` |
| 복잡도 판단, 분석 보고서, 수정 계획, 코드 수정 | `claude-sonnet-4-6` (메인) |

서브에이전트는 **요약된 결과만 반환**합니다. raw JSON, 파일 전체 내용 전달 금지.

---

## 실행 순서

### 1단계: 티켓 조회 + 링크 탐색 + 복잡도 판단

티켓 번호를 파싱하고, 티켓 본문과 댓글의 **링크까지 전부 열어서** 내용을 파악한 뒤 복잡도를 판단합니다.

#### 1-A. 티켓 조회 (Haiku 서브에이전트)

- Atlassian MCP(`getJiraIssue`)로 조회. MCP 없으면 `~/.claude/config.json`의 baseUrl + apiToken으로 REST API 직접 호출
- 반환: 제목, 버그 설명, 재현 절차, 에러 메시지, 언급된 파일/API/함수명, 키워드 5개, **현재 티켓 상태(status)**
- **링크 목록도 함께 추출**: 본문과 댓글에 포함된 모든 URL
- **티켓 상태가 `Done` / `Resolved` / `완료`인 경우 즉시 아래 경고를 출력하고 사용자 확인을 받습니다:**

```
⚠️  이 티켓은 이미 완료 상태입니다 (상태: Done)

그래도 분석을 진행할까요?
[Y] 계속 진행  [N] 취소
```

Y를 선택한 경우에만 이후 단계를 진행합니다.

#### 1-B. 링크 컨텍스트 수집 (Haiku 서브에이전트, 1-A와 병렬 실행 불가 — 1-A 완료 후 실행)

1-A에서 추출한 링크를 **전부 열어서** 내용을 확인합니다.

**Confluence 링크** (`*.atlassian.net/wiki/*`):
- `getConfluencePage` MCP로 페이지 내용 조회
- 핵심 내용 요약: 변경된 API 필드, deprecated 항목, 새 엔드포인트, 마이그레이션 가이드

**외부 링크** (GitHub, 사내 문서 등):
- `WebFetch`로 내용 조회
- 핵심 내용 요약

**반환 형식 (요약만):**
```
링크 컨텍스트:
1. [링크 제목] (Confluence/외부)
   - 핵심 내용: ...
   - deprecated 항목: ... (있을 경우)
   - 새 엔드포인트/필드: ... (있을 경우)
   - 수정에 필요한 정보: ...
```

> 링크가 없거나 열 수 없으면 이 단계는 건너뜁니다.

#### 1-C. 복잡도 판단 (메인 모델 — 티켓 요약 + 링크 컨텍스트 기반)

아래 신호가 **하나라도** 있으면 → **Fast Lane**
- 에러 메시지, 스택 트레이스가 포함됨
- 특정 파일명, 함수명, API 엔드포인트가 언급됨
- 댓글 또는 링크 문서에 수정 방향이 명시됨
- 링크 문서에서 deprecated API 또는 새 API 스펙이 확인됨

위 신호가 **전혀 없으면** → **Collaborative Lane**

---

### Fast Lane (빠른 자동 처리)

> 이슈 위치가 명확한 경우. 사용자 개입 없이 자동으로 진행합니다.

#### F-1단계: 정밀 탐색 (병렬, Haiku 서브에이전트 3개 동시 실행)

> 1단계에서 수집한 링크 컨텍스트(새 API 필드, deprecated 엔드포인트 등)를 탐색 키워드에 반드시 포함합니다.

**서브에이전트 1 — 에러/API 기반 탐색**
- 티켓의 에러 메시지, API명, 함수명 + **링크 문서에서 확인된 새 필드명/엔드포인트**로 Grep 검색
- deprecated API가 확인된 경우 → 해당 API를 현재 코드에서 사용하는 위치 탐색
- 반환: 일치하는 파일:라인 목록 (최대 5개), 각 1줄 요약

**서브에이전트 2 — Git 히스토리 탐색**
- `git log --oneline --all --grep="[티켓번호]"` 실행
- 관련 파일 최근 변경 이력 확인
- 반환: 관련 커밋 목록, 최근 변경 파일 (최대 3개), **티켓번호가 포함된 커밋이 있으면 `이미 수정된 커밋 있음` 플래그 포함**

**서브에이전트 3 — 연관 파일 추적**
- 서브에이전트 1이 찾은 파일의 import/dependency 1단계만 추적
- 반환: 직접 연관 파일 목록 (최대 3개), 각 1줄 요약

**서브에이전트 실패 처리:**

각 서브에이전트는 실패 시 자동으로 **최대 2회 재시도**합니다.
2회 재시도 후에도 실패한 에이전트가 있으면 아래를 출력하고 사용자 선택을 받습니다:

```
⚠️  일부 탐색에 실패했습니다

실패한 항목:
- [에이전트 이름]: [실패 원인]  (예: Git 히스토리 탐색 — .git 디렉토리 없음)

[이어서 진행]  확보된 정보만으로 분석을 계속합니다 (결과가 불완전할 수 있습니다)
[처음부터 다시]  전체 분석을 재시작합니다
```

- **이어서 진행** 선택 시 → 실패한 에이전트의 결과를 "확인 불가"로 표시하고 나머지 결과로 F-2단계 진행
- **처음부터 다시** 선택 시 → 1단계부터 재시작

#### F-2단계: 이미 수정됐는지 확인 후 수정 계획 제시

탐색 결과를 바탕으로 **수정 계획을 제시하기 전에** 아래 두 가지를 먼저 확인합니다.

**① Git 히스토리에 이미 수정된 커밋이 있는 경우:**

```
⚠️  관련 커밋이 이미 존재합니다

커밋: a1b2c3d [티켓번호] - 수정 내용 요약
날짜: 2026-03-20

현재 코드에 이 수정이 반영되어 있는지 확인하겠습니다.
```

→ 해당 커밋의 변경 내용을 현재 코드에서 실제로 검증합니다.
- 코드에 이미 반영됨 → "이미 수정되어 있습니다. 추가 수정이 필요하지 않습니다." 출력 후 종료
- 코드에 반영 안 됨 (예: revert됨) → "커밋은 있지만 현재 코드에 반영되지 않았습니다." 안내 후 수정 계획으로 진행

**② 탐색한 코드가 실제로 문제가 있는지 검증:**

수정 계획을 제시하기 전, 해당 파일/라인을 실제로 읽어서 아래를 확인합니다:
- 티켓에서 언급한 문제가 현재 코드에 실제로 존재하는가?
- 이미 다른 방식으로 처리되어 있지는 않은가?

현재 코드가 이미 올바르게 처리되어 있다면:
```
코드를 확인한 결과, 해당 문제가 이미 처리되어 있습니다.

위치: src/path/file.ts (line 42)
현재 코드: [실제 코드]

이유: [왜 이미 올바른지]

그래도 수정이 필요하신가요? [Y] 계속  [N] 종료
```

**③ 근본 원인인지 증상인지 판별:**

문제가 존재하는 경우, 수정 계획 진행 전에 이 위치가 진짜 원인인지 확인합니다:

- **이 위치가 잘못된 값을 직접 생성하는가?**
  → 근본 원인 → 수정 계획 진행

- **이 위치가 다른 곳에서 넘어온 값을 단순히 사용하는 구조인가?**
  → 증상 가능성 → 호출부(caller) 추가 탐색 (최대 3depth)

  **caller 추적 규칙:**
  - 1depth: 현재 파일을 호출하는 파일 탐색
  - 2depth: 1depth 파일을 호출하는 파일 탐색
  - 3depth: 2depth 파일을 호출하는 파일 탐색
  - 각 depth에서 원인 확인되면 → 해당 파일을 수정 계획에 포함하고 진행

  **3depth 이후에도 불명확한 경우:**
  ```
  ⚠️  호출 체인을 3단계까지 추적했지만 근본 원인을 특정하지 못했습니다.

  추적한 경로:
  - 1depth: [파일명]
  - 2depth: [파일명]
  - 3depth: [파일명]

  [현재 위치에서 수정]  증상 위치에서 방어적 처리로 수정합니다 (근본 원인은 아닐 수 있음)
  [파일 직접 지목]  의심되는 파일이나 함수를 알려주시면 바로 확인합니다
  [분석 종료]  분석을 중단합니다
  ```

---

이 두 확인을 통과한 경우에만 수정 계획을 제시합니다:

```
## [티켓번호] 수정 계획

### 원인
[한 문장으로 요약]

### 수정할 파일
1. src/path/file.ts (line 42-58)
   - 변경 내용: [구체적으로]
   - 이유: [왜]

### 예상 사이드 이펙트
- [영향받을 수 있는 부분]

### 실행할 테스트
- [테스트 명령어]

위 계획대로 진행할까요?  [Y] 진행  [N] 취소  [E] 수정
```

- **Y / "응" / "ㅇㅇ"** → F-3단계
- **N** → 종료
- **E / 수정 요청** → 계획 수정 후 재제시

#### F-3단계: 스타일 분석

코드를 수정하기 전, 수정 대상 파일과 인접 파일에서 프로젝트의 코딩 스타일을 파악합니다.

**분석 항목 (Haiku 서브에이전트):**
- 변수/함수 네이밍 규칙 (camelCase, snake_case, PascalCase 등)
- 함수 구조 패턴 (early return, guard clause, try/catch 위치 등)
- 주석 스타일 (JSDoc, 인라인 주석, 없음 등)
- import 구조 (named export, default export, 절대경로/상대경로)
- 에러 처리 방식 (throw, return null, Result 패턴 등)

**반환:** 스타일 요약 (5줄 이내). 수정 시 이 스타일을 반드시 따릅니다.

#### F-4단계: 두 에이전트 토론 후 코드 수정

단일 에이전트가 아닌, **구현 에이전트**와 **리뷰 에이전트** 두 관점으로 최적의 수정안을 도출합니다.

---

**[구현 에이전트 — 1차 수정안 제안]** (Sonnet)

- 수정 파일의 **관련 라인만** Read (파일 전체 읽기 금지)
- F-3단계 스타일 분석 결과를 반드시 준수
- 최소 범위 수정안 작성
- 제안 이유 명시 (왜 이 방식인지)

---

**[리뷰 에이전트 — 비판적 검토]** (Sonnet)

구현 에이전트의 수정안을 아래 기준으로 검토합니다:

| 검토 항목 | 확인 내용 |
|-----------|-----------|
| 정확성 | 이슈를 실제로 해결하는가? |
| 의존성 영향 | 수정된 함수·타입을 import하는 파일이 있다면 호환되는가? |
| 타입 안전성 | 파라미터·반환 타입이 호출부와 호환되는가? |
| 사이드 이펙트 | 다른 동작을 깨뜨리지 않는가? |
| 스타일 일관성 | 프로젝트 코딩 스타일을 따르는가? (F-3 결과 기준) |
| 비동기 처리 | async/await·에러 핸들링 패턴이 기존 코드와 일치하는가? |
| null 전파 | optional chaining·null 체크가 호출 체인 전체에 적용됐는가? |
| 엣지 케이스 | null, 빈 값, 예외 상황을 처리했는가? |
| 최소 수정 원칙 | 불필요한 변경이 없는가? |

**결과:**
- 문제 없음 → 최종 수정안 확정
- 개선 필요 → 구체적인 수정 의견 제시 → 구현 에이전트가 반영

> 토론은 최대 2라운드. 2라운드 후에도 합의 안 되면 리뷰 에이전트 의견을 우선합니다.

---

**[최종 수정 적용]**

- 합의된 수정안을 Edit으로 적용
- 완료 후 1줄 요약 출력

#### F-5단계: 영향 범위 분석 (Haiku 서브에이전트)

코드 수정 완료 후, 수정된 함수·컴포넌트명으로 **실제 영향받는 파일을 탐색**합니다.

**탐색 방식:**
1. 수정된 함수/컴포넌트/타입명으로 Grep → import하거나 직접 호출하는 파일 목록 추출
2. 각 파일에서 해당 호출부 라인만 Read (파일 전체 읽기 금지)
3. 시그니처 변경 여부, 호출 패턴 불일치 여부 확인
4. 2depth (1depth 파일을 import하는 파일)는 목록만 나열하고 분석은 생략

**출력 형식:**
```
## 영향 범위 분석

### 직접 영향 (1depth)
- src/screens/TradeInfo.tsx:42 — getUserInfo() 호출, 반환값 구조 변경으로 주의 필요
- src/hooks/useUser.ts:18    — 동일 함수 사용, 변경 없이 호환 가능 ✅

### 추가 확인 권장 (2depth)
- src/components/UserCard.tsx — useUser 훅 사용, 배포 후 빌드로 확인 필요
```

> 수정된 함수가 외부에서 사용되지 않으면 "영향 범위 없음"으로 기록하고 넘어갑니다.

#### F-6단계: 수정 리뷰 보고서 출력

수정 완료 후 반드시 아래 형식으로 리뷰 보고서를 출력합니다:


### Collaborative Lane (협업 처리)

> 이슈 위치가 불명확한 경우. 사용자의 힌트로 탐색 범위를 좁혀가며 진행합니다.

#### C-1단계: 1차 탐색 + 질문

키워드로 빠른 1차 탐색을 수행한 뒤, **구체적인 질문**으로 범위를 좁힙니다.

1차 탐색 (Haiku 서브에이전트):
- 티켓 키워드로 Grep + 최근 변경 파일 확인
- 반환: 유력 후보 파일 목록 (최대 3개) + 확신도(높음/중간/낮음)

탐색 결과를 보여주고 **예/아니오로 답할 수 있는 질문** 1~3개를 합니다:

```
## 1차 탐색 결과

유력 후보:
1. src/services/careService.ts — 간병 서비스 관련 로직 (확신도: 중간)
2. src/screens/TradeInfo.tsx — 거래정보 화면 (확신도: 낮음)

아직 확신이 없어서 몇 가지 여쭤볼게요:

Q1. 이 버그가 특정 사용자에게만 발생하나요, 모든 사용자에게 발생하나요?
Q2. 거래정보 화면에서 사용하는 API 이름이나 엔드포인트를 알고 있나요?
Q3. 최근 이 화면 관련 작업을 한 팀원이 있나요?
```

질문은 **범위를 좁히는 데 직접적으로 도움이 되는 것**만 선택합니다.
사용자가 답하면 → C-2단계로 이동합니다.

#### C-2단계: 힌트 기반 정밀 탐색

사용자의 답을 반영해 탐색 범위를 좁혀서 다시 검색합니다 (Haiku 서브에이전트).

- 힌트에서 새로운 키워드/파일명/API명 추출
- 정밀 Grep + 의존성 1단계 추적
- 반환: 재탐색 결과

결과에서 **위치 확신도가 높으면** → Fast Lane F-2단계로 합류해서 수정 진행
**여전히 불명확하면** → C-1단계 반복 (최대 3라운드)

3라운드 후에도 못 찾으면:
```
3번 탐색했지만 정확한 위치를 찾지 못했습니다.
지금까지 확인한 내용:
- [탐색한 파일 목록]
- [제외할 수 있는 영역]

직접 수상한 파일이나 함수를 알려주시면 바로 확인하겠습니다.
```

### 커밋 & 푸시 흐름

수정 완료 후 사용자가 커밋을 요청하면 아래 형식으로 커밋합니다.

**커밋 메시지 형식:**
```
[티켓번호] - 이슈 제목

- 수정 내용 1
- 수정 내용 2
- 수정 내용 3
```

**예시:**
```
[ABEH-1234] - 거래정보 화면에서 API 응답 파싱 오류 수정

- api.ts: getUserTradeInfo 응답 필드명 수정 (tradeName → tradeTitle)
- TradeInfo.tsx: undefined 접근 방지를 위한 optional chaining 추가
```

커밋 완료 후 반드시 사용자에게 물어봅니다:
```
커밋이 완료되었습니다.
푸시할까요? [Y] 푸시  [N] 나중에
```

- **Y / "응" / "ㅇㅇ" / "푸시해줘"** → `git push` 실행
- **N** → "나중에 git push로 직접 푸시하세요." 안내 후 종료

**푸시 실패 처리:**

푸시 실패 시 **1회 자동 재시도**합니다.
재시도도 실패하면 에러 메시지를 분석해 원인과 해결 방법을 안내합니다:

| 에러 패턴 | 원인 | 안내 |
|-----------|------|------|
| `rejected ... non-fast-forward` | 원격 브랜치에 로컬에 없는 커밋 존재 | `git pull --rebase` 후 재시도 안내 |
| `403` / `Permission denied` | 원격 저장소 쓰기 권한 없음 | 저장소 권한 확인 또는 SSH 키 설정 안내 |
| `Could not resolve host` | 네트워크 오류 | 네트워크 상태 확인 후 `git push` 직접 실행 안내 |
| `upstream` 관련 에러 | 원격 브랜치 연결 안 됨 | `git push -u origin [브랜치명]` 명령어 제시 |

```
⚠️  푸시에 실패했습니다 (재시도 포함 2회)

원인: [에러 패턴 분석 결과]
해결 방법: [위 표 기준 안내]

직접 해결 후 git push를 실행하거나, 해결 방법을 적용해드릴까요?
[직접 적용]  [건너뜀]
```

> 커밋과 푸시는 별도 권한 확인 없이 자동으로 실행합니다.
